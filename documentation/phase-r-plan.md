# Phase R — UX Refinements

## 1. What This Feature Does

Phase R is a focused UX refinement pass addressing four real-usage pain points surfaced by the parent operator:

1. **Cadence removal + prominent trigger** — the autonomous proposal loop (Phase J) added a scheduled background proposer that fires every 10/30/60 seconds. The parent never uses it and always manually fires the trigger instead. Remove the cadence loop entirely and make the manual trigger button the obvious primary action on the Play tab.

2. **Spoken text character limit** — the kiosk "Read Me" button speaks the full `step.body`, which can be several sentences long. Children with short attention spans tune out before it finishes. Add a parent-configurable character limit so the spoken version is brief; the full text remains visible on screen.

3. **Q&A gating** — children hit Next repeatedly without engaging with the activity. Add an optional `question` field to template steps; when set, the child kiosk shows the question and hides the Next button until the parent taps "Good answer" or "Skip" in the parent live-activity panel.

4. **Activity search** — parents cannot find past activities to replay them. Add a search surface covering both the history of past run activities and the template catalog, with a "Play again" / "Try this" re-trigger button on each result.

## 2. Existing Context

- **Activity loop** (`src/toybox/api/activities.py`, `src/toybox/activities/models.py`): activities are proposed → approved → running; each step advance is gated by `If-Match-Version`. The step shape (`ActivityStep`) is serialized from `activity_steps` rows and pushed to the child kiosk via WebSocket.
- **Settings pattern**: household-scoped scalar settings follow a uniform module shape (`src/toybox/core/<name>.py`: VALID set + DEFAULT + get/set; `src/toybox/api/<name>_settings.py`: GET/PUT router; migration seeds the row). Transcript retention and play_target_depth are canonical examples.
- **TTS path**: child kiosk's `ReadMeButton` (`frontend/src/child/components/ReadMeButton.tsx`) calls `speak(text, profile)` where `text` is `previewStep?.body` passed from `StepCard`. The truncation hook goes here.
- **Cadence stack**: `core/play_cadence.py` (the asyncio loop), `core/play_cadence_seconds.py` (setting module), `api/play_cadence_seconds_settings.py` (router), `core/proposed_ttl.py` (TTL sweep coupled to cadence). All started in `main._metrics_lifespan`; router registered in `app.py`.
- **Template step shape**: `Step` in `src/toybox/activities/models.py` is the template-time definition; `ActivityStep` is the runtime shape persisted to `activity_steps`. Both are Pydantic models; `ActivityStep` columns are serialized by `_row_to_response` in `activities.py`.
- **TypeScript codegen**: `frontend/src/shared/types.ts` is auto-generated; run `python tools/gen_types_ts.py` after any Pydantic model change. Drift is a pre-commit hook failure.
- **Frontend PIN gate**: the parent UI is PIN-gated; runtime reviewers (`--reviewers full/runtime`) cannot authenticate. All steps use `--reviewers code`.

## 3. Scope

**In scope:**
- Full removal of cadence background loop + proposed_ttl sweep (both are no-ops when cadence=0; proposed_ttl is meaningless without cadence)
- TriggerButton restyled as a prominent primary CTA (pre-Phase J style)
- Parent-configurable spoken text character limit (preset segmented control, client-side truncation to word boundary)
- Q&A gating: optional `question` field on activity steps, child kiosk hides Next until parent approves/skips
- Activity search: past runs + template catalog; re-trigger from results

**Explicitly out of scope:**
- Whisper auto-grade for Q&A (noted for future Phase; no STT-based answer evaluation here)
- Offline template mode for adventures (Phase T)
- Backend `spoken_text` field generated by Claude alongside `body` (future enhancement; R2 truncates client-side)
- Fixed-duration proposed TTL as a cadence replacement (can be added in a follow-up if stale suggestions accumulate)
- Cadence migration rollback or data migration (the 0012 settings row is left in place; no data loss)

## 6. Design Decisions

**Remove proposed_ttl entirely.** The TTL sweep (`proposed_ttl.py`) computes TTL as `cadence × 3`. When cadence is removed, there is no natural TTL multiplier. The alternatives are (a) fixed-constant TTL (e.g., 2 hours) or (b) removal. We remove it — manually-triggered suggestions should persist until the parent explicitly dismisses them; an auto-eviction of a carefully-chosen suggestion would be hostile. If stale-suggestion accumulation becomes an issue in practice, a fixed-TTL sweep can be added in a follow-up without schema changes.

**Client-side truncation for spoken text.** Adding a `spoken_text` column to `activity_steps` (generated by Claude alongside `body`) is the architecturally correct approach but requires schema migration, generator changes, and template updates. For R2, we truncate client-side: `ReadMeButton` receives a `limit` prop (0 = off); truncation snaps to the nearest word boundary at or below `limit` characters and appends `…` if truncated. The full `body` remains visible on screen. This is reversible — a future phase can add backend `spoken_text` without touching R2's truncation logic.

**Q&A approval is parent-only; no child dismissal.** The child kiosk shows the question and a "Waiting for parent…" placeholder. Only the parent can approve or skip. Giving the child a "skip" button defeats the purpose of the gate. The parent's "Skip" button resolves the gate without logging a negative signal — it's a neutral "move on" action.

**Q&A is per-step opt-in via template field.** Not every step needs a question. Authors add a `question` string to specific template steps. Steps without the field behave identically to today. The gating logic is in the advance endpoint: if the current step's `question IS NOT NULL` and `question_approved IS NULL`, advance returns 409 with code `question_pending`. The parent calls `POST /api/activities/{id}/approve-question` (body: `{result: "approved" | "skipped", version: N}`) to resolve.

**approve-question emits a WS envelope.** After the `approve-question` endpoint updates `question_approved`, it must broadcast an `activity.state` WebSocket envelope (same shape as the advance handler emits) so the child kiosk learns the gate is resolved and can show the Next button. Without this push, the child would remain stuck on "Waiting for parent…" until the next polling cycle or page reload. The WS broadcast is the same `_broadcast_activity_state(conn, activity_id, db)` call (or equivalent) already used by the advance handler — R3 must wire it into the approve-question endpoint as well, and R3's Done-when must verify the child kiosk transitions without a manual reload.

**approve-question bumps `activities.version` and returns `{version: N}`.** Every mutation that changes what the child kiosk renders must bump the `version` column so the If-Match-Version contract stays coherent. A parent who approves a question, then immediately taps Next, sends `If-Match-Version: N`; if approve-question didn't bump version, `N` is stale and the advance returns a spurious 409. The approve-question endpoint must increment `activities.version` in the same transaction as the `question_approved` update. The response body is `{"version": <new_version>}` (HTTP 200) — a minimal envelope that lets the parent UI update its local version counter without a separate GET. On advance gate failure the response is `{"detail": "question_pending"}` (HTTP 409), matching the FastAPI `HTTPException` pattern used throughout `activities.py`.

**Activity search uses LIKE scan (not FTS).** Full-text search (SQLite FTS5) is the right long-term answer but adds schema complexity. A `summary LIKE ?` scan over the 20-activity practical history is fast enough and requires no migration. Template catalog search scans the in-memory loaded template map (already loaded at startup). Upgrade to FTS is a straightforward follow-up.

**Search results are re-trigger friendly.** Every result (past activity or template) includes `template_id`. The re-trigger button POSTs `{ intent: "request_play", slot: "freeplay", template_id: <id> }` to `POST /api/activities/propose`. The existing propose handler already accepts `template_id` to pin a specific template; the generator short-circuits the random template selection when it's set.

**Search UI is a persistent input above PlayQueueList, not a new sub-tab.** The Play tab already has 5 sub-tabs (Phase O1); a 6th crowds the tab bar on iPad portrait. Instead, a compact search input sits pinned above the activity queue list when the Play tab is active. Typing into it filters/replaces the queue view with `SearchPanel` results; clearing the input restores the queue. No additional tab navigation is needed.

## 8. Risks and Open Questions

| Item | Risk | Mitigation |
|---|---|---|
| Cadence removal breaks existing DB users | Settings row 0012 stays in DB; reading it will fail silently if any code survives removal | Delete all readers in the same step; grep for `play_cadence_seconds` at step end |
| proposed_ttl removal means stale suggestions accumulate | A parent who triggers many suggestions and never dismisses them will see a long queue | Queue is capped at `play_target_depth` (eviction still runs on new proposals); parent can dismiss manually |
| Q&A gating: template authors don't know about the question field | Zero templates use it until authors add it | The field is fully optional; existing templates unaffected; the plan step adds a sample to at least one branching template as a smoke check |
| Search LIKE scan: `summary` is a JSON blob, not free text | LIKE on a JSON blob may miss results or return false positives | `summary` includes `title` in a predictable position; searching `%"title": "%<q>%"` targets the title field; a follow-up can add FTS if needed |
| Client-side TTS truncation: word-boundary algorithm may feel choppy | A sentence like "Wizard says…" truncated at 50 chars mid-sentence sounds abrupt | Truncation snaps to a word boundary and adds `…`; with default 150 chars ~1.5 sentences fit cleanly; parent can tune |
