### Step 1 — Cross-cutting finding (pre-existing, NOT an R3 regression) → issue #244

An adversarial reviewer pass + orchestrator independent verification surfaced a real enforcement gap that R3 did **not** introduce. R3's scope **design is correct**: `post_advance` is `{parent, child}` (activities.py:3742), `post_approve_question` is `{parent}`-only (activities.py:4098), and `TokenScope` defines both `parent` and `child` — i.e. a *child*-scope token can advance/read but must not clear the gate. The gap is that the production **child kiosk currently authenticates with a PARENT-scope token** (`POST /api/auth/parent`); the child-token pairing flow (`POST /api/auth/pair`) is **deferred to Phase D Step 20** and never called in the frontend (documented at `frontend/src/child/api.ts:6-13`). So today the kiosk's own credential *can* call `approve-question` — the gate is enforced by the child-route UI (no approve button) rather than by the credential boundary. Per `security.md` "UI is not a control."

- **Severity:** architectural HIGH / practical LOW for the target ages (4, 6) — exploitation requires manual API calls with the kiosk token; the child route hides the approve button.
- **Secondary (low):** `GET /api/activities/{id}` (`{parent, child}`) renders future template step `body`s for `proposed`/`approved` activities via `_render_template_plan_steps` (preview), visible to any token in pre-running states. The gated `question` field is NOT emitted on that path and lazy-insert (G2) means the next running-step row isn't persisted, so the question itself does not leak. Pre-existing G2.5 preview behavior, not R3.
- **Fix direction:** land Phase D Step 20 (child-scope kiosk token); then `approve-question`'s existing parent-only scope becomes a real credential boundary with **no R3 code change** needed. Tracked in **issue #244**. a7ebf84 remains verdict **OK** because the R3 diff is correct as written.

**Adversarial bypass result:** No bypass found. `args.host` is a single source — its `--host` default is `os.environ.get("TOYBOX_HOST", DEFAULT_HOST)` (main.py:138), and the SAME `args.host` value is passed to BOTH `check_bind_safe(args.host, ...)` (1236) AND `uvicorn.run(app, host=args.host, ...)` (1263). There is no divergence where the guard checks one host and uvicorn binds another. `TOYBOX_HOST=0.0.0.0` flows through `args.host` and is checked (documented in the main.py module docstring and operator/recovery.md:94). No flag combination, arg ordering, or env var lets the backend bind a non-loopback host while skipping the guard; the only way to allow a `0.0.0.0`/LAN bind is to have a real parent PIN set (`pin_set=True`), which is the intended invariant.

The single highest-priority concern — a bind-without-PIN bypass — does not exist: the guard runs unconditionally before the app is served, checks the exact host uvicorn binds, fails closed on a missing PIN or unopenable DB, and the launcher/.cmd add no bypass path. No code fix applied; the only change in this worktree is this markdown findings doc, which cannot move the green baseline (2323 pytest / 738 vitest), so the suites were not re-run.
