# Phase H — Parent UX revamp (tabs + global banned themes)

> **Scope:** Phase H build plan — replace the parent app's panel-toggle nav with a two-level tabbed shell (top tabs Play / Kids & Toyboxes / Settings; sub-tabs inside each), and promote `banned_themes` from a per-child CSV field to a single household-scoped setting. No gameplay or activity-engine changes; this is a UI restructure plus one schema migration that makes the runtime's already-UNION-across-children behavior the documented model. Carries the per-step `**Problem:**/**Type:**/**Issue:**/**Flags:**/**Files:**/**Status:**` shape that `/build-phase` parses. Sequenced after Phase G (DONE 2026-05-10); independent of Phase E (local model) — both can proceed in parallel since H touches the parent UI + one settings key, and E touches generation. Top-level overview is in [../plan.md](../plan.md).

## What this feature does

Today the parent app at [frontend/src/parent/App.tsx](../../frontend/src/parent/App.tsx) is a single-page surface with a row of toggle buttons (`child profiles`, `toys`, `rooms`, `operator`, `transcripts`) that conditionally render one panel below the header at a time. The `Trigger play` button sits separately in the same row. There is no grouping, no default view, and no concept of which surface goes with which workflow.

Phase H reorganizes the surface into three top-level tabs, each with 2-3 sub-tabs:

- **Play** — `Play Ideas` (default; the existing Trigger button + the proposed/running activity surfaces) | `Transcription`
- **Kids & Toyboxes** — `Toys` (default) | `Children` | `Rooms`
- **Settings** — `Settings` (default; toggles + global banned themes) | `Stats` (metrics snapshot)

The `Operator` panel is split: toggles (listening mode, mic mute, image-gen mode) move to Settings, the metrics snapshot moves to Stats. The `Trigger play` button is renamed to `Play Ideas` and lives inside the Play tab alongside the SuggestionCard / ActivityPanel that today sit at the bottom of the App. Tab selection (top + sub) persists in `localStorage` so an operator returning to the page lands where they left off.

In parallel, `banned_themes` moves from a per-child CSV field on `ChildProfile` to a single key in the existing `settings` table. The escalation pipeline at [src/toybox/core/escalation.py:629](../../src/toybox/core/escalation.py#L629) already UNIONs banned themes across all children before sending them to Claude, so the runtime semantics don't actually change — the data just stops being modeled per-child. Migration 0009 unions whatever per-child data exists into the new global key, then drops the `children.banned_themes` column. The existing preset picker (`bannedThemePresets.ts`) survives wholesale; it just now writes into a global textarea instead of a per-child one.

## Vocabulary and conventions

**UX terms:**
- **Top tab** — one of the three section selectors at the top of the parent app (Play / Kids & Toyboxes / Settings). Exactly one selected at a time.
- **Sub-tab** — a pill-style selector inside a section (e.g., Play Ideas / Transcription within Play). Exactly one selected per section; selection is independent per section.
- **Default sub-tab** — the sub-tab pre-selected on first load (and on first switch into a section if no localStorage entry yet): Play → Play Ideas; Kids & Toyboxes → Toys; Settings → Settings.
- **Activity surface** — collective term for `SuggestionCard` + `ActivityPanel`. Both render only inside the Play Ideas sub-tab; switching to another top tab during a running activity hides them (state is preserved in App.tsx; visibility is purely a render-time gate).

**Tab state shape** (held in App.tsx, hydrated from localStorage at mount):

```ts
type TopTab = "play" | "kids-toyboxes" | "settings";
type PlaySubTab = "play-ideas" | "transcription";
type KidsToyboxesSubTab = "toys" | "children" | "rooms";
type SettingsSubTab = "settings" | "stats";

interface TabState {
  top: TopTab;                    // default "play"
  play: PlaySubTab;               // default "play-ideas"
  kidsToyboxes: KidsToyboxesSubTab; // default "toys"
  settings: SettingsSubTab;       // default "settings"
}
```

**localStorage keys** — one key per dimension, namespaced `toybox.parent.tabs.*`:
- `toybox.parent.tabs.top` → `"play" | "kids-toyboxes" | "settings"`
- `toybox.parent.tabs.play` → `"play-ideas" | "transcription"`
- `toybox.parent.tabs.kids-toyboxes` → `"toys" | "children" | "rooms"`
- `toybox.parent.tabs.settings` → `"settings" | "stats"`

Reads are defensive: any value not in the union falls back to the default (handles stale entries from older versions or hand-edited storage). Writes happen on every selection change.

**Banned-themes data shape**:
- **Storage** — `settings` table, key `banned_themes_global`, value `TEXT` (NULL means "no household ban list"). Comma-separated, mirrors the existing per-child CSV format so the preset picker doesn't change shape.
- **API** — `GET /api/settings/banned-themes` returns `{themes: string | null}`; `PUT /api/settings/banned-themes` accepts `{themes: string | null}`. Both are gated by the parent-token middleware. Idempotent.
- **Migration** — 0009 unions all non-NULL `children.banned_themes` values: split each on commas, trim whitespace, lowercase, dedupe across all children, sort, re-join with `, `. Writes the result to `settings.banned_themes_global` if non-empty (otherwise leaves the key absent). Then rebuilds `children` without the `banned_themes` column (SQLite requires CREATE TABLE...AS SELECT + DROP + RENAME for column removal).

**Build orchestration** (skill names referenced throughout this doc):
- `/build-step` — runs one step end-to-end in an isolated git worktree: implements per the step's Problem statement, runs the named reviewers, opens a PR.
- `/build-phase` — orchestrates a multi-step phase by reading this plan doc, dispatching `/build-step` per row in the Steps table, and posting progress to the umbrella + per-step GitHub issues.
- `/repo-sync` — mints/updates the umbrella + per-step GitHub issues from this plan and backfills the `**Issue:**` numbers in each step block.

**Build-step flag conventions** (per the `/build-step` skill — same as Phase G):
- `--reviewers code` — four code-quality reviewers; default for autonomous-build operating mode.
- `--reviewers full` — code + runtime reviewers.
- `--isolation worktree` (default) — agent works in a temporary git worktree.

**Step `Type:` taxonomy:** `code` (autonomous `/build-step`), `operator` (manual procedure executed by a human, not a build agent), `wait` (long-wall-clock observation step that just waits for an external signal — e.g., a soak run completing).

**Test stack acronyms** (used in done-when commands):
- **Vitest** — Jest-compatible unit test runner used by the React frontend (`npm run test`).
- **Playwright** — headless-browser end-to-end / UI test framework (`npm run test:ui`).
- **Testing Library** — React component testing helpers (`screen`, `getByRole`, etc.) used inside Vitest tests.

**UAT** — User Acceptance Testing; manual operator-driven walkthrough on real hardware (here, the iPad parent app) verifying the phase's deliverables hang together end-to-end.

**Out:**
- Tab keyboard shortcuts (left/right arrow, number keys) — defer; not requested
- Mobile/responsive layout for the parent app on phones — out of scope; iPad parent-app UX is in scope but no portrait work
- Per-child banned themes overrides on top of the global list — explicitly dropped per the "merge into global, drop field" decision; if the user wants a per-child override later, it's a follow-on phase
- Auto-focus to Play tab when a new suggestion arrives — explicitly not selected; operator stays on whatever tab they were on
- Persistent activity-running banner above the tabs — explicitly not selected; activity surface is Play-tab-only
- Path-aware activity history in the Stats sub-tab — Stats keeps the existing snapshot only; activity-history aggregates are deferred
- Global banned-themes API consumed by anything outside the parent app — escalation pipeline reads it server-side after migration, but no other client surface needs it
