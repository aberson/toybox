# Phase I — Transcript retention + display refresh

> **Scope:** Phase I build plan — add a configurable retention window to the `transcripts` table (preset durations 1m / 3m / 5m / 10m / 15m, default 1 minute) backed by a periodic backend sweep + filter-on-read; in the parent UI, remove the per-row delete affordance (wipe-all stays) and add a local-timer fade-out animation as rows cross their expiry. No changes to the transcription pipeline itself — STT capture, persistence, and the live WS push all stay as Phase H left them; this phase changes only retention and display. Organized for maximum autonomous parallelism: after I1 lands, I2 (backend) runs in parallel with I3 (frontend Settings UI + plumbing); I4 (frontend list refresh) sequences after I3; I5 (UAT) fans in. Carries the per-step `**Problem:**/**Type:**/**Issue:**/**Flags:**/**Files:**/**Status:**` shape that `/build-phase` parses. Sequenced after Phase H (DONE 2026-05-10); independent of Phase E (local model). Top-level overview is in [../plan.md](../plan.md).

## What this feature does

Today the parent app's Transcription sub-tab (Play → Transcription, mounted by Phase H) renders an indefinite cursor-paginated list of every transcript ever captured. Each row carries a per-row delete button; a separate "wipe all" button at the top of the panel clears the table after a PIN re-confirm. The `transcripts` table accumulates forever — there is no TTL, no sweep, no expiration. This works for an audit log but mismatches the "live feed of what the kid just said" usage pattern the operator has settled into.

Phase I makes transcripts ephemeral by default:

- **A new household-scoped setting `transcript_retention_seconds`** lives in the existing `settings` table. Default value `60`. Valid values: `{60, 180, 300, 600, 900}` (1m / 3m / 5m / 10m / 15m). No "Forever" option (explicit user decision — retention is always finite).
- **Backend sweep** runs every 10s in the production lifespan, hard-deleting transcripts whose `ended_at` is older than `now - retention`. Rows with `ended_at IS NULL` (in-flight utterances) are skipped — they become eligible the moment the pipeline writes `ended_at`.
- **Filter-on-read** is applied to `GET /api/transcripts` and `GET /api/transcripts/search` using the same cutoff, so API responses are consistent with the swept view even between sweep ticks.
- **Frontend Settings UI** gets a new preset segmented control under SettingsPanel for picking the retention value. Mirrors the existing `ImageGenModeToggle` pattern: optimistic local state, PUT-on-click, no debounce.
- **Frontend list UI:** the per-row delete button is removed (the `deleteTranscript` API client method goes with it — it's not referenced anywhere else). The wipe-all button + PIN modal stay verbatim. Each row computes its own `expires_at = ended_at + retention` and a 1s tick checks for expired rows; expired rows get a CSS class that triggers a 600ms opacity-fade + height-collapse transition, then are removed from local state.
- **Retention plumbed through React props** from App.tsx (single fetch on mount) down to both SettingsPanel (writes) and TranscriptsManager (reads). When SettingsPanel writes a new value, App's local state updates, propagating into TranscriptsManager — rows re-evaluate their `expires_at` and any that are newly past expiry fade out in real time. No WS broadcast (single-parent kiosk model; React state is the cross-component channel).

## Vocabulary and conventions

**Retention terms:**
- **Retention seconds** — the integer value stored in `settings.transcript_retention_seconds`. One of `{60, 180, 300, 600, 900}`. Default `60`.
- **Cutoff** — `now() - retention_seconds`, expressed as an ISO-8601 timestamp string in the **exact same format the pipeline writes** to `ended_at` (verified by I1; pinned in all retention helpers and tests). Computed fresh per sweep tick and per API read.
- **Expiry** — the wall-clock time at which a given transcript becomes eligible for deletion: `ended_at + retention`. Computed client-side (for fade animation timing) and server-side (for sweep + filter-on-read) using the same retention value.
- **In-flight transcript** — a row with `ended_at IS NULL`, written by the pipeline at utterance start and updated when the utterance ends. Excluded from sweep and from the read filter so a still-being-spoken utterance never disappears mid-render.
- **Sweep cadence** — 10 seconds. The lifespan task wakes every 10s, computes the cutoff once, and runs a single `DELETE FROM transcripts WHERE ended_at IS NOT NULL AND ended_at < ?` statement. With min retention = 60s, a row lives at most 60-70s before being deleted; at max retention = 900s, 900-910s. Drift is bounded by the cadence.

**Settings storage shape:**
- **Key:** `transcript_retention_seconds` (TEXT key in the existing `settings` table — value stored as the stringified integer, matching how `pin_hash` and `image_gen_mode` store text).
- **Default fallback:** `current_retention_seconds` returns `60` (`RETENTION_SECONDS_DEFAULT`) in **three** cases: (a) row missing, (b) value unparseable as int, (c) value parseable as int but **not** in `RETENTION_SECONDS_VALID`. Cases (b) and (c) log a warning at WARNING level with the offending value truncated to 64 chars (mirrors `image_gen_mode.py`'s truncate-the-blob pattern). Legacy DBs or hand-edited corrupt values both resolve cleanly to the default.
- **Validation:** `set_retention_seconds` rejects values outside the canonical set with `ValueError`; the API translates this into HTTP 422 via explicit `HTTPException` in the route handler.

**API shape:**
- **GET `/api/settings/transcript-retention`** — returns `{"seconds": int}`. No auth (consistent with `GET /api/settings/image-gen-mode`, which is also read-public). The retention value is not a secret; matching the public-read pattern keeps the API surface uniform.
- **PUT `/api/settings/transcript-retention`** — accepts `{"seconds": int}`, gated by `RequireScope(TokenScope.PARENT)`. Idempotent. Validates against the canonical set; 422 with `{"error": "invalid_retention", "valid": [60, 180, 300, 600, 900]}` on a bad value.

**No WS broadcast on retention change** — single-parent kiosk model. The setting is fetched once on App.tsx mount and threaded via React state to both SettingsPanel and TranscriptsManager. When SettingsPanel PUTs a new value and gets a 200 back, it calls the App-provided `onRetentionChange` callback; App updates state; both consumers re-render. A second tab open on the same parent app would see a stale value until reload — acceptable for v1.

**Initial-fetch failure behavior:** if `getTranscriptRetention()` on App.tsx mount rejects, log to console at WARN level and retain the optimistic default of `60`. The user sees the 1m button highlighted in SettingsPanel; the next PUT will succeed and overwrite the optimistic state. No error toast (the rest of the parent app shrugs off settings-fetch failures the same way — the listening mode toggle uses an optimistic seed from `/api/metrics` with the same posture).

**Fade animation shape:**
- **Trigger:** a 1s `setInterval` inside TranscriptsManager iterates the local items list, computes `expires_at = ended_at + retention` for each row, and marks any row with `expires_at <= now` as `fading`.
- **Style:** the `fading` row gets inline styles `{ opacity: 0, maxHeight: 0, marginTop: 0, marginBottom: 0, paddingTop: 0, paddingBottom: 0, overflow: 'hidden', transition: 'opacity 600ms ease, max-height 600ms ease, margin 600ms ease, padding 600ms ease' }`. **`max-height: 0`** (not `display: none`) so the height-collapse transition plays smoothly; setting `display: none` would skip the transition.
- **Removal:** a `setTimeout(600)` queued at fade-trigger time removes the row from local state after the transition completes. If retention changes mid-fade (operator goes from 1m → 15m while a row is fading), the in-flight `setTimeout` still fires and removes the row — acceptable because the row was already past its old expiry. Newly-expired rows on the new retention are picked up by the next 1s tick.
- **Malformed `ended_at` handling:** if `new Date(row.ended_at).getTime()` is `NaN` (defensive — shouldn't happen since the pipeline writes valid ISO, but a corrupt or in-flight-but-non-null value could), `Number.isNaN` check skips the row on this tick + logs to `console.warn` once per row id. Row neither fades nor accumulates spurious fade attempts. A unit test covers this case.
- **Pause-on-hover:** **not implemented** in v1 (not requested). If a parent wants to read a long transcript before it fades, they bump the retention to 15m or copy the text.

## Scope

**In:**
- **Migration 0010** (pure SQL, idempotent): seeds `settings('transcript_retention_seconds', '60')` via `INSERT OR IGNORE` and adds `CREATE INDEX IF NOT EXISTS idx_transcripts_ended_at ON transcripts(ended_at)`. Both are idempotent — re-running on a populated DB is a no-op. The index keeps the I2 sweep + filter-on-read fast as the table churns. No schema rewrite, no data migration beyond the seed row.
- **Backend helper module** `src/toybox/core/transcript_retention.py`: `current_retention_seconds(conn) -> int` + `set_retention_seconds(conn, value: int) -> int`. Mirrors `core/image_gen_mode.py` shape — same `_SETTINGS_KEY` constant + `SELECT value FROM settings WHERE key = ?` + `INSERT OR REPLACE` upsert. Validation of the canonical set lives in the helper, not the API layer. Default-on-missing-row + default-on-unparseable + default-on-out-of-set, all behind a single fall-through that returns `RETENTION_SECONDS_DEFAULT`.
- **Backend API module** `src/toybox/api/transcript_retention_settings.py`: `APIRouter(prefix="/api/settings", tags=["transcript_retention_settings"])` exposing GET + PUT per the Vocabulary shape. PUT is `RequireScope(TokenScope.PARENT)`-gated; GET is unauthenticated (mirrors `GET /api/settings/image-gen-mode`). Registered in `src/toybox/app.py` alongside the other settings routers.
- **Backend sweep helper** in `src/toybox/core/transcript_retention.py` (same module as the get/set helpers — single concern, single file): `sweep_expired_transcripts(conn, now: datetime) -> int` runs one DELETE pass and returns `cursor.rowcount` (the deleted row count). Pure function, easy to unit-test with a controllable `now`. The loop driver `async def run_transcript_sweep_loop(conn_factory, *, interval_seconds: float = 10.0, clock: Callable[[], datetime] = lambda: datetime.now(timezone.utc)) -> None` wraps it in a `while True: await asyncio.sleep(interval); sweep(...)` with broad exception capture per tick (logs at ERROR + continues) and a `_logger.info("swept %d expired transcripts", count)` line on ticks where count > 0.
- **Lifespan helper** `transcript_sweep_lifespan(app)` in `src/toybox/app.py`: `@contextlib.asynccontextmanager` that spawns the loop driver as a task on enter and cancels + awaits it on exit. Modeled exactly on `image_gen_worker_lifespan`.
- **Production composition** in `src/toybox/main.py`: the new lifespan is composed into the existing stack alongside `image_gen_worker_lifespan` + audio capture + metrics. Smoke/test-only lifespans do not need the sweep.

**Out:**
- **"Forever" / "never expire" retention option** — explicitly declined per user decision. Maximum retention is 15 minutes.
- **Free-text or slider retention input** — explicitly declined; preset segmented control only.
- **Per-child or per-session retention overrides** — not requested; single household-wide setting.
- **Pause-on-hover** to prevent fade while the operator is reading — not requested; sufficiently mitigated by bumping retention.
- **Re-export of expiring rows to a CSV log file** — hard delete is hard delete. No archival side-channel.
- **Recover-from-trash** — impossible by design; rows are physically deleted.
- **Search across expired rows** — impossible by design. Search results shrink consistently with the list view; that's the explicit contract, not a bug.
- **WS broadcast** of retention-setting changes to other parent app instances — single-parent kiosk model.
- **Migration that truncates existing transcripts on first run** — the natural behavior of the new sweep cleaning up older rows is sufficient. Operator gets a heads-up via the I2 PR notes + a required pre-merge DB backup per I2's done-when.
- **Per-row "extend by 5 minutes" button** — not requested; out of scope.
- **Changes to the transcript STT pipeline** — `audio/pipeline.py` is unchanged. Insertion semantics, confidence scoring, WS push, intent escalation — all unchanged.

## Risks

- **R1: 1-minute default + apply-immediately means existing transcripts vanish on first sweep.** When I2's sweep starts running on a production DB with months of accumulated rows, the first tick deletes every row older than 60 seconds — which is essentially all of them. The wipe is irreversible (hard delete, no archival side-channel). **Mitigation:** I2's done-when makes the pre-merge `data/toybox.db.pre-i2.bak` backup a required step, and the I2 PR description must call it out in bold at the top. I5's pre-flight (step 0) verifies the backup exists. If the operator wants to preserve history beyond the backup, they can bump retention to 15m before I2 merges (deferring the wipe by 15 minutes is the maximum buffer) or accept the loss — Phase I's premise is that the table is a live feed, not an archive.
- **R2: Sweep loop death silently leaves transcripts accumulating.** If the sweep task throws an unhandled exception and the loop terminates, the table grows without bound and the filter-on-read becomes the only check. **Mitigation:** the loop driver wraps each tick in `try / except asyncio.CancelledError: raise / except Exception: _logger.exception(...)`, so any non-cancellation error logs at ERROR with a full traceback and the loop continues to the next tick. The lifespan helper itself crashing on enter would prevent the server from starting (FastAPI propagates lifespan errors) — at least loud failure, not silent. I2's test (f) in `test_transcript_retention_sweep.py` exercises the exception-resilient loop path. A future operational improvement could expose sweep-tick metrics; out of scope here.
- **R3: Concurrent reads + deletes.** Operator scrolling the list while the sweep deletes rows. **Mitigation:** none needed structurally. The `transcripts` table has no FKs pointing INTO it (only its own FK to `sessions`), so the delete is a single non-cascading statement against an indexed column. Read queries use their own connection. SQLite's WAL mode (already enabled per project convention) handles concurrent readers + a writer cleanly. The worst case is the operator briefly seeing a row in the API response that the sweep deleted milliseconds later — at most one render cycle of staleness, swallowed by the next 1s frontend tick.
- **R4: Reducing retention mid-session triggers a "wave of fades."** Operator drops from 15m → 1m with 30 visible rows; on the next 1s tick, 29 rows simultaneously enter the fading state and fade out together over 600ms. **This is correct behavior** — not a risk, more a callout that the UX is intentional. The visual is acceptable per the planning conversation; if it ever proves jarring, a future PR could stagger the fade timings.
- **R5: Search use case effectively dies.** With max 15-minute retention, the substring-search endpoint operates on at most a few minutes of data. The search affordance in TranscriptsManager remains functional but its utility shrinks significantly. **Mitigation:** none in this phase; keep the search UI for the cases where short-window search is still useful (catching a specific recent utterance). If search across longer windows becomes critical, a future phase could add an opt-in "archive on expiry" pattern — but that's explicitly out of scope here.
- **R6: ISO format drift between pipeline and sweep.** If the pipeline writes `ended_at` in a format different from what the sweep helper computes for the cutoff, lexicographic string comparison can give the wrong answer even when the underlying instants are well-ordered. **Mitigation:** I1's build step inspects `_insert_transcript` and pins the exact format in a `transcript_retention.py` constant; I2's sweep + tests + filter-on-read all use the same constant. The I2 sweep test (a) uses pipeline-format fixture rows (or inserts via the real `_insert_transcript`) to lock the contract. If the pipeline ever changes its format, the sweep tests catch the mismatch before it ships.
- **R7: Fade-CSS interaction with virtualization or large lists.** The fade applies `max-height: 0` to a row container — if the list ever moves to a virtualized renderer (e.g., react-window), this animation strategy needs to coexist with the virtualizer's own height calculations. **Mitigation:** TranscriptsManager is not virtualized today (cursor-paginated load-more, no virtualization), and with max 15-minute retention the visible row count will stay small. Defer.
- **R8: Cross-step merge ordering between I3 and I4.** I3 adds `retentionSeconds: number` (required) to TranscriptsManagerProps; I4 adds the consumption logic in the function body. Diffs are textually disjoint (interface block vs. function body) so they should auto-merge. But if I4 lands BEFORE I3, App.tsx isn't passing the now-required prop and typecheck fails. **Mitigation:** the dependency graph explicitly sequences I4 after I3. `/build-phase` honors `Depends on: I3` in the I4 step. I4's worktree should branch from a base that already includes I3's interface change (either I3 merged to master, or the I4 agent branches from I3's branch directly).

## Decisions log (made during planning)

- **Expiry behavior:** hard delete from SQLite. Row is physically removed; nothing to recover. Storage stays bounded; matches the "live feed, not archive" intent. Soft-hide (filter-on-read only) was considered but rejected because it would let the DB grow unbounded.
- **Retention setting UI:** preset segmented control with 5 buttons: 1m / 3m / 5m / 10m / 15m. No "Forever" option. No free-text input. Slider was considered and rejected (too imprecise for the 5-value space). Dropdown was considered and rejected in favor of the more scannable button row (matches SettingsPanel's other toggles).
- **Default retention:** 1 minute (60s). Aggressive but matches the live-feed intent. Operator can opt up to 15m if they want longer visibility.
- **First-deploy behavior:** apply retention immediately on first sweep after migration lands. No grandfathering, no one-time wipe in the migration. Existing rows that are older than 1m at sweep time are deleted naturally by the first sweep tick. R1 captures the mitigation; the backup requirement moved into I2's pre-merge gate (not I5) so the operator backs up before the destructive change ships.
- **Sweep cadence:** 10 seconds. Tight enough that the DB stays in close sync with the UI (max 10s of drift past a row's true expiry); loose enough to not thrash the DB connection. Alternatives considered: 30s (too loose for the 60s minimum retention), 1s (unnecessary churn).
- **WS broadcast on retention change:** none. Single-parent kiosk model. State is plumbed through React props (App.tsx → SettingsPanel + TranscriptsManager). A second parent tab would see a stale value until reload — acceptable for v1.
- **In-flight transcripts** (rows with `ended_at IS NULL`): excluded from sweep and from the read filter. They become eligible only when the pipeline writes `ended_at`. Prevents a still-being-spoken utterance from disappearing mid-render.
- **Per-setting module convention:** mirror `core/image_gen_mode.py` + `api/image_gen_settings.py` exactly. New helper at `core/transcript_retention.py`, new router at `api/transcript_retention_settings.py`. Sweep helper lives in the same `transcript_retention.py` module — single concern (retention).
- **Per-row delete removal:** full removal of the UI button + the `DELETE /api/transcripts/{id}` backend endpoint + the `deleteTranscript` frontend API method. No soft hide, no opt-in keep. Wipe-all + PIN modal retained verbatim.
- **Filter-on-read in addition to sweep:** yes, both. Sweep keeps the DB bounded; filter-on-read keeps API responses consistent with the UI's local computation between sweep ticks (max 10s of drift would otherwise show as a transient "row should be gone but is still here" between the backend sweep and the frontend's local tick).
- **Frontend fade implementation:** local 1s timer + inline-style opacity/height transition + setTimeout to remove from state after 600ms. Pure client-side; no WS coordination, no server "about to expire" signal. Simpler and avoids extra wire traffic. `max-height: 0` chosen over `display: none` to allow the height-collapse to animate; `display: none` would skip the transition entirely.
- **Out-of-canonical-set integer in stored setting:** treated the same as unparseable — log a WARNING + fall back to `RETENTION_SECONDS_DEFAULT`. Defensive for hand-edited DBs or future migration mistakes. Tested explicitly in I1.
- **Malformed `ended_at` on the frontend:** skip the row on this tick + log once via `console.warn` (tracked via `useRef<Set<string>>` to avoid spam). Row stays in the DOM until the operator manually clears via wipe-all. Tested explicitly in I4.
- **Initial-fetch failure on App.tsx mount:** log to `console.warn` + retain optimistic default `60`. No toast (matches the other settings' posture of "the optimistic seed is safe; the next user action will succeed or surface the error then").
- **GET endpoint is public** (no auth): consistent with `GET /api/settings/image-gen-mode`. The retention value is not a secret; keeping reads public keeps the API surface uniform.
- **Search results shrink consistently with the list view:** an explicit contract, not a bug. The substring search endpoint filters by the same retention cutoff as the list. R5 documents the reduced utility; if it ever becomes critical, a future "archive on expiry" pattern can be added without changing the v1 semantics.
- **I3/I4 split** (interface declaration in I3, consumption in I4): chosen to keep diffs textually disjoint inside `TranscriptsManager.tsx`. I3 owns App.tsx + SettingsPanel + the new component + the prop interface. I4 owns the TranscriptsManager body + `api.ts` deleteTranscript removal. Two parallel worktrees can be developed concurrently as long as I4 branches from a base that includes I3's interface change. The merge order is I3 → I4 (enforced by `Depends on` in the dep graph).
- **I2/I3 parallelism:** I2 is entirely `src/toybox/` + `tests/`; I3 is entirely `frontend/`. Zero file overlap. Both depend on I1 only, so once I1 lands `/build-phase` can dispatch them concurrently. Their wall-clock cost overlaps fully.
- **Operating mode:** bundled UAT per Phase G/H convention. Frontend-touching steps run `--reviewers code` (no per-step Playwright UI evidence); visual verification is the I5 operator pass.
- **Format:** new phase plan doc + GitHub issues + `/build-phase` execution. Treats Phase I with the same ceremony as F.5 / G / H / iPad-Kiosk.
