# Phase W — Play Depth (dials, STT grading, dynamic adventures + boss fights)

## 1. What This Feature Does

Phase W is a combined feature pass covering six not-yet-built items the parent operator
asked for, ordered cheapest-first so value ships before the risky engine work. It adds
(1) two **true-stub** parent settings dials (involvement, game complexity) that persist a
value but wire to no behavior yet; (2) a real **linear ↔ non-linear games toggle** that
filters template selection; (3) **Whisper/STT answer-grading** for Q&A question steps with
a parent tolerance toggle, extending the Phase R parent-tap approval path; and (4) a
**dynamic adventure engine** that generates the next story beat from the child's choices +
recently-heard speech — hybrid (Claude when online via the capability gate, offline
template-assembled fallback) — with **interactive boss fights** emitted as a special climax
beat by that same engine. It is built because the existing catalog activities are static and
the parent wants play that adapts to the child in the moment.

## 2. Existing Context

A fresh-context model needs these load-bearing facts (all confirmed against source, not docs):

- **Settings convention (canonical: `spoken_text_limit`).** Each household-scoped scalar
  setting is a quartet: `core/<name>.py` (a `*_VALID` frozenset + `DEFAULT` + `get_<name>` /
  `set_<name>` with `ValueError`→HTTP-422), `api/<name>_settings.py` (a `GET` unauthenticated
  + `PUT` parent-scope router under prefix `/api/settings`), a migration that seeds the
  `settings` row, and a parent-UI segmented control mirroring `SpokenTextLimitControl.tsx` /
  `TranscriptRetentionControl.tsx`. Verified in
  [`spoken_text_limit.py`](../../src/toybox/core/spoken_text_limit.py) +
  [`spoken_text_limit_settings.py`](../../src/toybox/api/spoken_text_limit_settings.py). There
  are 10 existing settings routers under `src/toybox/api/*_settings.py`.
- **Settings store.** A single `settings(key TEXT PRIMARY KEY, value TEXT)` table; values are
  TEXT and parsed in the `get_*` helper (`int(raw)` with fallback-on-unparseable). No WS
  broadcast — single-parent kiosk model; the next `App.tsx` mount re-fetches.
- **Next migration number is `0024`** (highest on disk is
  [`0023_activity_step_question.sql`](../../src/toybox/db/migrations/0023_activity_step_question.sql)).
- **Q&A gating (Phase R Step R3) already exists.** `activity_steps.question TEXT NULL` +
  `question_approved INTEGER NULL` (migration 0023); `question_approved` is `1`=approved,
  `2`=skipped, `NULL`=pending. The advance handler returns `409 {"code":"question_pending"}`
  when `question IS NOT NULL AND question_approved IS NULL`
  ([`activities.py:3868`](../../src/toybox/api/activities.py)). `POST /api/activities/{id}/approve-question`
  resolves it, bumps `activities.version`, and emits an `activity.state` WS envelope
  (handler at [`activities.py:~4100`](../../src/toybox/api/activities.py)). `Step.question` /
  `ActivityStep.question` are Pydantic fields in
  [`models.py`](../../src/toybox/activities/models.py).
- **STT is already in the stack.** [`audio/stt.py`](../../src/toybox/audio/stt.py) wraps
  faster-whisper; the home passive mic streams VAD-gated chunks → transcripts persisted with
  retention (default 1m). The child kiosk (iPad PWA) is display-only — it does **not** capture
  audio. Any "hear the answer" path must read the recent passive-transcript window, not kiosk
  audio.
- **Capability gate.** [`core/capability.py`](../../src/toybox/core/capability.py) `compose_capability`
  + the live `is_capable()` decide whether a Claude call may run; on `False` the system uses the
  offline path with a stable `capability_reason`. Every Claude call goes through it
  (invariant #4 in `plan.md`).
- **Activity step shape + advance.** Steps persist to `activity_steps`; `kind` on the runtime
  `ActivityStep` is a **free-form string** (existing values: `text`, `fork`, `song`, `joke`,
  `reward`, `element_microgame`) — new runtime kinds need **no migration**. The advance handler
  (`post_advance`, [`activities.py:3736`](../../src/toybox/api/activities.py)) lazily inserts the
  next step on each `POST /api/activities/{id}/advance`; branching steps post
  `{"choice_index": N}`. This lazy-insert seam is where the adventure engine plugs in.
- **Template selection.** [`generator.py`](../../src/toybox/activities/generator.py) `generate()`
  loads one JSON library per intent, filters by `buckets`, sorts by `template_id`, and picks one
  with a seeded RNG. Branching templates carry `choices`; linear ones don't. This is where the
  linearity toggle filters.
- **Frontend PIN gate.** The parent UI is PIN-gated; runtime reviewers cannot authenticate, so
  every code step uses `--reviewers code` (per
  `feedback_buildstep_pin_gate_blocks_ui_evidence`).
- **TS codegen.** `frontend/src/shared/types.ts` is generated by `python tools/gen_types_ts.py`;
  drift is a pre-commit failure. Any new Pydantic response model requires a codegen run.

## 3. Scope

**In scope:**
- Two true-stub settings dials (parent involvement, game complexity): persisted value + parent
  UI segmented control, **no behavior wired**.
- Linear ↔ non-linear games toggle: persisted boolean + parent UI + real wiring into
  `generator.generate()` template filtering (linear → exclude branching templates).
- STT Q&A answer-grading: a per-household tolerance setting (`off | lenient | strict`), an
  optional `expected_answer` field on question steps, and an auto-grade path that resolves the
  R3 gate from the recent passive-transcript window (offline fuzzy match; online Claude judge via
  the capability gate) — falling back to the existing parent-tap when grading is `off` or no
  confident match is found.
- Dynamic adventure engine: generate the next beat from prior choices + recent transcript;
  hybrid Claude-online / template-assembled-offline; persisted as runtime `activity_steps` via
  the existing lazy-insert advance seam.
- Interactive boss fights: a `kind="boss_fight"` climax beat the adventure engine emits, reusing
  the Phase K `boss_mini_boss` / `big_bad_boss` roles; kiosk renders a boss-fight step variant.
- A pipeline smoke gate + an iPad UAT covering all six features.

**Explicitly out of scope:**
- Wiring the involvement / complexity dials to any behavior (true stubs this phase; a follow-up
  phase consumes them).
- Kiosk-side microphone capture (grading reuses the home passive mic only).
- A local fine-tuned model for adventure generation (Phase E territory; W4 uses Claude-or-template).
- Multi-session / persistent adventure state across activities (each adventure is one activity).
- New persona art or boss sprites beyond reusing existing toy/persona sprites (image-gen is
  Phase P/U/V).
- Voice answer-grading confidence tuning beyond the two-tier `lenient/strict` tolerance.

## 5. New Components

- **`core/parent_involvement.py` / `core/game_complexity.py`** — stub settings. Each:
  `*_VALID = frozenset({"low","medium","high"})`, `DEFAULT = "medium"`, `get_*`/`set_*` exactly
  mirroring `spoken_text_limit.py` (TEXT value, fallback-on-unparseable, `ValueError`→422). No
  consumer reads them this phase — that is the point.
- **`core/game_linearity.py`** — `GAME_LINEARITY_VALID = frozenset({"linear","nonlinear"})`,
  `DEFAULT = "nonlinear"` (preserves today's branching behavior). Read by `_do_propose`.
- **`core/qa_grading.py`** — `QA_GRADING_VALID = frozenset({"off","lenient","strict"})`,
  `DEFAULT = "off"` (preserves today's parent-tap-only behavior). Plus a pure
  `grade_answer(transcript_window: str, expected: str, tolerance: str) -> bool` offline grader
  (normalize case/punctuation, token-overlap threshold per tolerance). Online grading is a
  separate Claude judge call made by the API layer through the capability gate; `grade_answer`
  is the offline fallback and the unit-testable core.
- **`core/boss_fights_enabled.py`** — boolean flag, `DEFAULT = True`, mirroring
  `jokes_enabled.py`.
- **`activities/adventure.py`** — the engine. `generate_next_beat(history, transcript_window,
  cast, *, online: bool, beat_index: int, boss_enabled: bool) -> GeneratedBeat`. When
  `online` (capability gate green) → Claude prompt that takes the choice history + recent speech
  and returns a beat `{body, choices?, kind}`. When offline → assemble a beat from the existing
  branching-template fragments / generic descriptors deterministically. Emits
  `kind="boss_fight"` as the climax when `boss_enabled` and `beat_index` crosses the climax
  threshold. Returns a typed `GeneratedBeat` the advance handler persists as an `activity_steps`
  row. The Claude call site lives in the API layer (gate + client injection) and delegates the
  prompt/parse to this module so the engine stays unit-testable without network.
- **5 parent-UI controls** — segmented buttons mirroring `SpokenTextLimitControl.tsx`
  (optimistic update + revert-on-error).

## 6. Design Decisions

**Stub dials persist only (chosen).** The involvement + complexity dials ship as full
settings quartets with no consumer. Rationale: it reserves the UX slot and the persisted value
so a later phase can wire behavior without a parent-facing change, and it lets the operator start
expressing preference now. Alternative (wire complexity → branching depth, involvement → Q&A
frequency) was rejected for this phase to keep W1 a mechanical, low-risk step; the wiring is a
clean follow-up.

**Linearity toggle filters template selection (chosen).** `linear` excludes any template whose
steps carry `choices`; `nonlinear` (default) preserves today's behavior. Implemented as a
`linear_only` filter in `generate()` rather than a separate template library, so the existing
1000+ catalog is reused and the toggle is one predicate. When the adventure engine (W4) is
active, the same toggle suppresses generated `choices` (linear adventures are a single spine).

**STT grading extends, never replaces, parent approval (chosen).** With `qa_grading=off`
(default) the behavior is exactly today's R3 parent-tap. With `lenient`/`strict`, on advance the
handler first attempts an auto-grade against the recent passive-transcript window: offline fuzzy
match, or a Claude judge when the gate is green. A confident match sets `question_approved=1`
(same column, same WS envelope, same version bump as approve-question) so the kiosk unblocks with
no parent tap. No confident match leaves the gate pending for the parent — grading only ever
*removes* parent taps, never blocks progress that the parent could previously unblock. Tolerance
controls the offline overlap threshold and the Claude judge's strictness instruction.

**Grading reads the home passive mic, not the kiosk (forced by hardware).** The kiosk is
display-only. The grader reads the transcript rows in a short window after the question became
current. This is best-effort: if the room mic doesn't hear the child, grading simply doesn't
fire and the parent resolves it. Documented as a risk, not a guarantee.

**Adventure generation is hybrid via the capability gate (chosen).** Online → Claude generates
the next beat from choices + speech (richest). Offline → deterministic template-assembled beat.
This mirrors every other Claude path in toybox (invariant #4) and means adventures never hard-fail
when the network is down. Alternatives rejected: Claude-only (breaks offline-first), template-only
(loses the "builds on spoken words" value the operator asked for).

**Adventures plug into the existing lazy-insert advance seam (chosen).** Rather than a new
activity lifecycle, an adventure is an ordinary activity with `activities.adventure=1`. The
advance handler already inserts the next step lazily; for adventure activities it calls
`adventure.generate_next_beat(...)` instead of reading the next template row, persisting the
result as a normal `activity_steps` row. This reuses optimistic concurrency, WS broadcast, the
reward-at-end path, and the kiosk renderer with minimal new surface.

**Boss fight is a climax beat, not a standalone activity (chosen).** A boss fight is
`kind="boss_fight"` emitted by the adventure engine at the climax (after a threshold of beats),
casting a Phase K `boss_mini_boss` / `big_bad_boss` role. It reuses adventure plumbing and the
role taxonomy; the kiosk adds one render variant. Alternative (standalone boss-fight activity
type with its own propose path) was rejected as more surface for the same payoff. Gated by the
`boss_fights_enabled` flag.

**Adventure generation is request-driven, not background.** Beat generation fires synchronously
inside `POST /api/activities/{id}/advance` (a user-invoked request that completes and returns).
There is **no** new daemon, scheduler, or always-on watcher — so the autonomous-behavior soak
requirement does not apply. It IS a producer→consumer pipeline (generate → persist → kiosk
render), so a smoke gate (W6) is required before UAT.

## 8. Risks and Open Questions

| Item | Risk | Mitigation |
|---|---|---|
| Home mic can't hear the child at the kiosk | STT grading silently never fires; parent thinks it's broken | Grading is best-effort and additive — `off` by default; ActivityPanel shows "listening…" and the parent approve/skip stays available; documented as best-effort in the control help text |
| Claude adventure beats may produce age-inappropriate or off-tone content | A generated beat could surface banned themes or scary content for a 4-yo | Reuse the existing banned-themes/global filter on generated beat text; clamp beat length to the spoken-text limit; offline fallback is template-sourced (already curated); boss fights gated by a flag |
| Adventure generation latency on advance | A Claude round-trip per advance makes the kiosk feel slow | Generate synchronously but cap with the existing client timeout; on timeout/gate-not-green, fall back to the offline template beat immediately (never block the child) |
| Free-form `kind` typo on a generated beat | An unknown kind renders blank on the kiosk | `StepCard` keeps a safe default render for unknown kinds; engine emits only the two documented new kinds; unit test asserts emitted kind ∈ known set |
| `generate()` signature change misses a caller | Linear toggle silently ignored on a propose path | W2 Done-when greps every `generate(` call site and updates each (code-quality.md §1) |
| Offline "builds on spoken words" is weak | Template-assembled beats can't truly incorporate arbitrary speech | Accept reduced fidelity offline (documented in Scope); the spoken-word richness is the online path's value; offline still threads the prior *choice* |
| Migration count (0024–0028) in one phase | A mid-phase failure leaves the DB partway forward (forward-only) | Each step ships exactly one migration and is independently testable; abort+preserve per invariant #10; steps ordered so each migration is self-contained |

**Open questions (resolve in plan-review if needed):**

  Defaulting to a fixed count this phase since the complexity dial is a stub.
