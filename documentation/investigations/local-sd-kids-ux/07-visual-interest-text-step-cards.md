# Visual interest for text-heavy step cards

**Tier:** 2
**Status:** Investigated
**Related:** [[03-activity-scene-backgrounds]], [[06-illustrated-adventure-mode]], [[17-visual-schedules-step-thumbnails]], [[02-kiosk-style-cohesion]]

## What it is
`StepCard.tsx` is the kiosk's universal step renderer for EVERY activity
type — text, fork, adventure_beat, boss_fight, song, joke, reward, element.
On a plain text/fork step it is a centered white card (`rgba(255,255,255,0.82)`,
`borderRadius: 20`) holding a step-counter line, an `<h1>` body, optional
cast sprites flanking the body row (`step-body-row`: left/right `ToyActionSprite`
columns), and a Next/Choice button. The richest non-text kinds already pull
visual furniture into the card (ElementCard above the body, the static
"⚔ Boss Fight ⚔" banner, RewardStep), but the everyday `text` and `fork`
steps — the bulk of play — are mostly a wall of bold text with at most a
flanking sprite.

This topic asks how to break up that text-heavy default with layout and
imagery so a plain step reads as lively rather than worksheet-like. It is the
generalization of [[06-illustrated-adventure-mode]] (which targets only
adventure beats) to ALL activity types, and the layout host for the backdrops
in [[03-activity-scene-backgrounds]] and the thumbnails in
[[17-visual-schedules-step-thumbnails]].

## Why it matters
Child B (4, pre-reader) cannot read the `<h1>` at all — a text-only card is
pure adult chrome to him; imagery is the only channel that carries meaning.
Child A (6, early reader) can read but tires of dense text and loses the "this is
play" feeling. Failure mode: the operator's "a little boring" verdict is
largely THIS card, because it's what's on screen 90% of the time. Over-correct
and you get the ransom-note problem ([[02-kiosk-style-cohesion]]) or a card so
busy the body text loses legibility against it.

## When to apply
- Any `text` / `fork` / `adventure_beat` step (the default render path).
- Steps with a single flanking sprite and lots of body text.
- Pre-reader sessions where the body is unreadable to the child.

## How to apply
Three cost tiers, cheapest first:

1. **Pure CSS / layout (zero generation, ship now).** Reuse the existing
   primitives: scale the flanking `ToyActionSprite` up on low-cast steps,
   add a persona-tinted accent strip or rounded illustration frame, vary the
   card's corner/shadow per persona to match S1 surface + Phase S gradients.
   The `idleBob` keyframe already makes resting sprites breathe — extend that
   gentle, flash-free vocabulary (slow drift, no opacity change) rather than
   add strobe. Honors `prefers-reduced-motion` like the existing CSS does.
2. **Reuse already-generated assets (no new GPU work).** Surface the toy's
   existing per-slot action sprites (`pointing`, `cheering`, `thinking`) as
   the contextual illustration instead of always `idle` — the slot is already
   threaded via `previewStep.action_slot`. This is free: the sprites exist on
   disk under `data/images/toy_actions/<toy_id>/<slot>.png`.
3. **New generated imagery (GPU, batch).** A small per-step illustration or
   card backdrop generated by the SD pipeline. On a single-worker GPU at
   4-step LCM (~seconds/image) this must be PRE-RENDERED at activity-approve
   time or overnight, never on-demand mid-step — the worker is FIFO and a
   live kid tap cannot wait. Defer the heavy lever to [[03-activity-scene-backgrounds]].

**Prototype:** start in tier 1 — add a `data-step-kind`-driven accent + bump
sprite size in `StepCard.tsx`, vitest-assert the new layout, iPad-UAT for
legibility. Safety: imagery rides the same negative-prompt + parent-preview
guardrails as all generation; layout-only changes carry none.

## Open questions
- Does a per-step backdrop hurt body legibility enough to need a guaranteed
  text scrim (overlaps [[03-activity-scene-backgrounds]])?
- Can tier-2 sprite-slot variety alone move the "boring" needle, deferring
  GPU work entirely? Cheapest test to run first.
- Pre-render budget: how many step illustrations can realistically batch at
  activity-approve time before the kid taps "I'm Ready"?
