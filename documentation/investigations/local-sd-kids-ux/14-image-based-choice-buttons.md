# Image-based choice buttons (branching)

**Tier:** 3
**Status:** Investigated
**Related:** [[05-coloring-page-line-art-mode]], [[15-vocabulary-concept-illustration]], [[03-activity-scene-backgrounds]]

## What it is
Branching steps (Phase G) render one `ChoiceButton` per entry in `step.choices`, each a text-only pill showing a free-text label like "Into the jungle" or "Up the cliff" (from `request_play.json`). A pre-reader (Child B, 4) cannot read those labels — the only non-text affordance today is the optional Phase-K `ClickableText` tap-to-speak, which still requires tapping to hear. Image-based choice buttons add a small generated thumbnail to each button so the picture itself communicates the branch: a jungle scene vs. a cliff scene, a dragon vs. a castle.

The thumbnail is a per-choice illustration of where that path leads, generated by the local SD pipeline (`pipeline.py`) and composited above or beside the label, not a sprite of a household toy.

## Why it matters
Pre-reader UX payoff is the whole point: branching is the most agentic moment in a kiosk activity — the kid steers the story — yet it is currently the moment most gated on reading. A picture makes the fork legible to Child B and lets Child A (6, early reader) confirm her guess. Failure modes: (1) ambiguous or generic thumbnails that all look alike defeat the purpose and may mislead a child into a path they didn't want; (2) style drift between buttons looks like a ransom note (see [[02-kiosk-style-cohesion]]); (3) latency — generating thumbnails on-demand at the fork would stall the single-worker GPU mid-activity.

## When to apply
- A step has `choices` (a fork). Linear steps need no choice art.
- The kid profile is a pre-reader, or `clickable_words` is on — gate behind a parent flag so early readers can keep text-only buttons.
- The branch destinations are visually distinct enough that a thumbnail disambiguates (jungle vs. cliff: yes; "be brave" vs. "be kind": no — falls back to [[15-vocabulary-concept-illustration]] / icon).

## How to apply
Pre-render the catalog, do not generate on-demand. The ~1360 templates across four intents are static JSON; every choice label is fixed text. An offline batch (like the Phase G 4-agent template soak, or the Phase U `batch_animate.py` pattern) can walk every template's `choices[].label`, generate one SD thumbnail per distinct label, and cache by a content hash of the label string so duplicate labels across templates share one image. Estimate: count distinct labels first (grep `"label"` across `branching/*.json`); likely far fewer than 1360×N once deduped. At 4-step LCM ~1-2s/image on the warm cached pipeline, a few thousand thumbnails is an overnight batch, not a runtime cost.

Generation: drop the IP-Adapter reference (no toy identity needed — these are scene/concept thumbnails), keep the `2D cartoon, simple shapes, clean lines` style suffix and the rembg pass for consistency with sprites. Render small (256px is plenty for a button). Cache to `data/images/` keyed by label hash; `ChoiceButton` gains an optional `thumbnailUrl` prop rendered above the (still-present) label.

Prototype: pick one template (`request_play_soak_pirate_01`), generate two thumbnails for its fork, hard-code the URLs into `ChoiceButton`, and iPad-check whether Child B picks the intended path without the label. Effort is low — the button already has clean local state and an existing `data-testid`.
